## 1. Executive Summary

The codebase is in **materially better shape than the 2026-03-03 audit and several explore-tool reports suggested**. Of the 11 CRITICAL findings raised in March, 7 are verified closed in source today, 1 is partially closed, and 3 require deeper verification or remain open. The major hardening work in commits `0a93cec` ("critical security audit"), `d99c31a` ("pro audit fixes"), and `be62fc2` ("per-country payment routing, KYC, invoice payments") landed cleanly: idempotency guards exist on reversals, multi-tenant scoping is wired on every financial table, webhook signature verification is timing-safe, CSP is enabled in production, and bill-payment atomic flows now populate every tracking column.

What is genuinely concerning is no longer at the per-handler level — it is **operational, observability, and platform-level**. A single one-click endpoint can purge the entire transactional database. The recurring-payment scheduler runs unconditionally on every container instance with no leader election. The infrastructure stack has a single NAT gateway, a single-AZ RDS, and one CDK stack shared across environments. The CI/CD pipeline ships to production with no test gate. Mobile lacks jailbreak detection, certificate pinning, and encrypted local storage. The frontend stores Cognito tokens in localStorage and validates forms by hand rather than reusing the Zod schemas already exported from `shared/`.

**Bottom line:** Spendly v4 is shippable today for the current scale (early customers, single-region, single-instance ECS); it is **not** ready for the multi-region, multi-instance, regulated-customer footprint described in the existing PRD. The remediation path is clearly bounded — see §9 — and most items are 0.5- to 2-day fixes.

**Severity headlines:**

- 4 CRITICAL findings (3 platform, 1 product-surface)
- 13 HIGH findings (mix of operational, security, mobile, frontend hardening)
- ~22 MEDIUM and LOW findings (data hygiene, observability, dev experience)

## 3. Backend Audit

### 3.1 Boot sequence and middleware order

[server/index.ts:1-150](../../server/index.ts) — The Express 5 app sets up in this order:

1. Process-level handlers for `uncaughtException` and `unhandledRejection` (logger.fatal → exit)
2. Required env-var guards (`DATABASE_URL` always; `COGNITO_USER_POOL_ID`, `COGNITO_CLIENT_ID`, `APP_URL` in production; explicit refusal of `_test_` payment keys in production)
3. `app.set('trust proxy', 1)` — correct for ALB-fronted deployment
4. Helmet with strict CSP in production (self + Stripe + Paystack + S3 + AWS), HMR-relaxed in dev
5. JSON body parser with `verify` callback that captures raw body for webhook signature verification
6. CORS — explicit origin allowlist in prod (`thefinanciar.com`, `www.thefinanciar.com`, `app.thefinanciar.com`, plus `APP_URL`); permissive in dev
7. CSRF protection on `/api` (`X-Requested-With` requirement)
8. Rate limiting on `/api` (5-tier from [server/middleware/rateLimiter.ts])
9. Request logger with correlation IDs via `pino`
10. Route mounting via `registerRoutes`
11. Global error handler (logs and returns sanitized 500)

This is a defensible production-ready setup.

### 3.2 Route surface

- **26 route modules** under `server/routes/` (24 domain + `index.ts` + `shared.ts`); a legacy `server/routes.legacy.ts` still exists with overlap and should be retired (see [AUD-BE-009](#7-risk-register))
- **~517 total endpoints** by raw `router.<verb>` count across `server/routes/*.ts` and the legacy file
- All financial routes use the triple-gate pattern: `requireAuth + requireAdmin + requirePin` for mutating operations; `requireAuth` only for reads

### 3.3 Persistence layer

| Domain | Tables (representative) |
|---|---|

| Money movement | `expenses`, `bills`, `invoices`, `payouts`, `payoutDestinations`, `fundingSources`, `payrollEntries`, `scheduledPayments`, `vendors` |
| Cards & accounts | `virtualCards`, `cardTransactions`, `virtualAccounts` |

Multi-tenant scoping is now consistent: every financial table has `companyId` with an FK to `companies.id` (mostly `onDelete: 'set null'` on read-side tables, `cascade` on tightly-bound joins like `companyMembers`). Index coverage on `companyId` is present everywhere I checked.

**Three notable structural items:**

1. **`transactions.walletTransactionId`** is declared at [shared/schema.ts:261](../../shared/schema.ts) without a Drizzle `.references()` call. The comment at lines 259–260 explains: forward references are not supported by Drizzle, so the FK is enforced at SQL level via migration. This works at runtime but means TypeScript-level joins through Drizzle's relation API don't see the link. Consider restructuring to a separate junction or using `pgTable` with deferred FK if Drizzle 0.40+ supports it.

2. **`bills` table now has approval-tracking columns** (`approvedBy`, `approvedAt`, `reviewerComments` at [shared/schema.ts:295-297](../../shared/schema.ts)) and bill-payment tracking (`paidBy`, `paidDate`, `paymentMethod`, `paymentReference`, `walletTransactionId`). The 2026-03-03 finding "no approval audit fields" is partly addressed for bills; expenses still need verification.

3. **The legacy `users` table** is still present for backward compatibility ([shared/schema.ts:194-200](../../shared/schema.ts) marked `@deprecated`). Plan removal once a migration confirms zero active reads.

### 3.4 Storage layer (`server/storage.ts`)

A 2,416-line god-object implementing the `IStorage` interface (~230 methods). Spot-checks of the four atomic operations:

| Operation | File:line | Properties |
|---|---|---|
| `atomicBillPayment` | storage.ts:1484-1548 | `FOR UPDATE` lock on wallet; balance check; debit walletTx + `bills` row updated atomically; **all 7 tracking fields populated** |
| `atomicCardFunding` | storage.ts:1550-1614 | `FOR UPDATE` lock on wallet AND on card; balance check; walletTx + card balance update atomic |
| `atomicWalletTransfer` | storage.ts:1616-1701 | `FOR UPDATE` on both source and dest wallets; balance check; debit + credit walletTx in one DB transaction; honors optional `exchangeRate` |
| `atomicReversal` | storage.ts:1703-1773 | `FOR UPDATE` lock on wallet; **idempotency guard** at line 1722-1725; reversal walletTx insert; original walletTx marked with `reversedAt + reversedByTxId` |

**Real concerns at the storage layer:**

- The four atomic ops only write `walletTransactions`, **not** `transactions`. The Transactions page on the client reads from `transactions` only. This means bill payments, card fundings, wallet transfers, and reversals do not appear in the user-facing transaction history. ([AUD-BE-002](#7-risk-register))
- The 2,416-line single class is hard to test, hard to onboard new engineers to, and concentrates change risk. Domain extraction is overdue ([LU-011](LOGIC_UPGRADE_PROPOSALS.md#lu-011-storage-layer-domain-split)).

### 3.5 Authentication and authorization

[server/cognito-verifier.ts](../../server/cognito-verifier.ts) + [server/middleware/auth.ts](../../server/middleware/auth.ts):

- AWS Cognito JWT verification via `aws-jwt-verify` library — kid caching is built in, JWKS auto-refresh handled by the SDK
- Verifies `tokenUse: 'id'`, issuer (Cognito user-pool URL), audience (`clientId`), signature, expiration
- Production fail-closed: if `COGNITO_USER_POOL_ID` is missing in prod, all `requireAuth` routes return 503
- Dev fallback: `x-admin-user-id` header for backward compatibility (must be removed before going truly multi-customer)
- Triple gate `requireAuth + requireAdmin + requirePin` is applied to financial mutations
- PIN check at [server/middleware/auth.ts] uses the financial rate limiter (3 failed attempts per minute)

### 3.7 Background jobs

[server/recurringScheduler.ts](../../server/recurringScheduler.ts):

- **Default interval: 1 hour** (`startRecurringScheduler(intervalMs: number = 3600000)` at line 275). The 24-hour figure cited elsewhere is incorrect.
- Three sub-jobs run in sequence: `processRecurringBills`, `processRecurringPayroll`, `processScheduledPayments`
- Dedup is **in-memory** via `allBills.some(...)` at line 50–53 — race-prone if two instances run concurrently
- **No leader election.** When the ECS service runs > 1 task (which it must, for HA), every instance executes the scheduler, creating duplicate bills and payroll entries.
- Uses **`console.log/console.error`** (lines 31, 70, 71, 126, 128, 157, etc.) instead of the structured `pino` logger that the rest of the app uses. Operations team will not be able to correlate scheduler events with request traces.
- The "FIX P4" comment at line 166 documents a real prior bug (balance leak when external transfer failed); the fix itself is correct. But it relies on a non-transactional sequence of `updateBalances → external API call → on error, refund updateBalances` — under crash mid-call this can still leak. A proper outbox pattern would be more durable.

This is the highest-impact open finding in the backend ([AUD-BE-001](#7-risk-register)).

### 3.8 Logging and observability

[server/lib/logger.ts](../../server/lib/logger.ts):

- Pino with `pino-pretty` in dev, JSON in prod — good
- PII redaction is **partially implemented**: `redact.paths` covers `req.headers.authorization`, `req.headers["x-transaction-pin"]`, `req.headers.cookie`. The `maskPII` function defines patterns for email/phone/card/CVV/NUBAN/BVN but is **not applied automatically to log payloads** — only available as a manually-invoked helper. The 2026-03-03 audit's claim of "no PII redaction" is wrong, but the redaction is incomplete.
- Correlation IDs propagate via `X-Correlation-Id` request → response — good
- Status-code-aware level routing (`>=500 error`, `>=400 warn`, `>=200 info`) — good
- **No APM** — no OpenTelemetry, no Honeycomb, no Datadog. Production p99 latency, error rates, and external-call traces are invisible. ([LU-013](LOGIC_UPGRADE_PROPOSALS.md#lu-013-observability))

### 3.9 Tests

11 unit-test files (~2,677 lines) under `server/__tests__/`, `shared/__tests__/`, plus 1 Playwright spec at `e2e/app.spec.ts`. The 353-passing claim in commit `4e79399` is plausible.

The integration tests are mock-heavy. There is no `testcontainers` Postgres; payment flows are exercised against in-memory mocks. For a fintech codebase this is a meaningful gap — production migrations and atomic operations under contention are precisely the things mocks cannot exercise. ([AUD-BE-005](#7-risk-register))

### 3.10 Migrations

[migrations/](../../migrations/) — 8 SQL files (`0000` through `0007`), linear, hand-written. The latest, [0007_add_security_constraints.sql](../../migrations/0007_add_security_constraints.sql), adds:

- `wallets_user_currency_unique` on `(user_id, currency)` — prevents duplicate wallets
- `wallet_transactions_reference_unique` on `(reference)` — DB-level idempotency
- Partial unique index on `transactions(reference) WHERE reference IS NOT NULL`

Migrations are run on container startup by [scripts/run-migration.cjs](../../scripts/run-migration.cjs), which is idempotent (auto-skips "already exists"/"duplicate key" errors). This is acceptable for the current scale; for stricter environments consider a proper migration table that tracks applied versions.

## 4. Frontend Audit

### 4.1 App skeleton

[client/src/App.tsx:1-300](../../client/src/App.tsx) — Wouter routing with three route classes:

- **Public** (10): `/`, `/login`, `/signup`, `/forgot-password`, `/terms`, `/privacy`, `/cookies`, `/onboarding`, `/admin-login`, `/auth/callback`
- **Protected** (~14): dashboard, transactions, expenses, bills, budget, cards, accounts, team, settings, analytics, reports, payroll, invoices, vendors
- **Admin** (~10): under `/admin/*`, gated by `AdminRoute` which reads sessionStorage + revalidates via `/api/admin/verify-session`

`AuthProvider` calls `checkAuthState()` on mount, hydrating Cognito session from localStorage (Cognito SDK auto-detects via the `CognitoIdentityServiceProvider.{clientId}.{username}.idToken` keys). Skip-to-main-content link is present for accessibility.

### 4.2 Page inventory (36 pages)

Grouped:

- **Auth (5):** login, signup, forgot-password, auth-callback, admin-login
- **Dashboard / core (4):** dashboard, transactions, analytics, reports
- **Money movement (8):** expenses, bills, invoices, virtual-accounts, pay-invoice (public), payroll, cards, vendors
- **Team (3):** team, onboarding, invite (public token route)
- **Admin (10):** admin, admin-users, admin-audit-logs, admin-organization, admin-security, admin-wallets, admin-payouts, admin-exchange-rates, admin-database, admin-login
- **Legal / misc (5):** privacy, terms, cookies, landing, not-found

### 4.3 State and data fetching

[client/src/lib/queryClient.ts](../../client/src/lib/queryClient.ts):

- TanStack Query v5 with `staleTime: 30s`, `refetchOnWindowFocus: false`, conservative retry (network errors only, max 2 attempts)
- Auth headers fetched fresh on every request via `getAuthHeaders()` (calls Cognito SDK for current ID token), plus `X-Company-Id` header for tenant scoping and `X-Requested-With: XMLHttpRequest` for CSRF
- Global error sanitizer (`sanitizeErrorMessage`) strips HTTP codes, SQL errors, and stack traces before display
- 401 handling: queries default to throwing; some queries opt into `on401: "returnNull"`
- **No optimistic updates anywhere.** Every mutation is pessimistic, which means slow networks produce visible latency on common paths (creating an expense, sending money). ([AUD-FE-007](#7-risk-register))

### 4.4 Forms and validation

The client does **not** import the Zod schemas exported from `shared/schema.ts` (via `drizzle-zod`'s `createInsertSchema`). Validation in pages like login, signup, expense-create, and bill-pay is hand-rolled with regex and length checks. Drift risk is real — the server-side schema can change without the client noticing.

### 4.5 Auth flow

[client/src/lib/cognito.ts](../../client/src/lib/cognito.ts) and [client/src/lib/auth.tsx](../../client/src/lib/auth.tsx):

- Uses `amazon-cognito-identity-js` (the legacy Cognito SDK, predates AWS Amplify v6). Works, but no longer the recommended path.
- ID token, access token, refresh token all live in `localStorage`. **An XSS hole anywhere in the app → all tokens exfiltrated.** ([AUD-FE-001](#7-risk-register))
- An additional `sms_id_token` is stored in localStorage for SMS-OTP login flows with no TTL — never expires until the user explicitly logs out. ([AUD-FE-002](#7-risk-register))
- OAuth (Google) flow: `/auth/callback` exchanges code at the Cognito token endpoint and stores tokens in localStorage manually (lines 237–263 of cognito.ts)
- Logout: `globalSignOut()` invalidates all tokens at Cognito, clears localStorage, calls `queryClient.clear()`. Good.
- Admin session: a separate flow stores adminUser in **sessionStorage** as plain JSON, then revalidates via `/api/admin/verify-session` on each admin route visit. The plain sessionStorage value is editable from DevTools (admin role can be self-asserted before revalidation), so the protection rests entirely on server-side revalidation. Acceptable but brittle.

### 4.6 Admin pages

`admin-database.tsx` is the most concerning. Despite the file name, it is **not** an arbitrary SQL execution surface — it is a single-button, hardcoded "Purge Database" UI calling `POST /api/admin/purge-database` with body `{ confirmPurge: "CONFIRM_PURGE", tablesToPreserve: [...] }`. The server endpoint at [server/routes/admin.routes.ts:223](../../server/routes/admin.routes.ts) wipes every transactional table after admin-role check.

Concrete risk: an attacker who phishes an admin's password (or a disgruntled internal admin) can wipe every customer's wallets, transactions, expenses, bills, virtual cards, payroll, invoices, vendors, and notifications with a single click. The server logs the action to `audit_logs` but as `userId: 'system'` — the actual admin's identity is not captured at the audit-log level. ([AUD-FE-003 / AUD-BE-003](#7-risk-register))

### 4.7 Stray code

- `admin-dashboard.jsx` at the **repo root** (75 KB, 1,521 lines, `.jsx` not `.tsx`, outside the Vite source tree). `grep` across the repo finds zero imports. Dead code — delete in a single line of work. ([AUD-FE-004](#7-risk-register))

### 4.8 Money formatting

The 2026-03-03 finding "Expenses, Bills, Payroll, Budgets missing currency selectors" is **closed**. All four pages now import `formatCurrencyAmount` from `shared/constants.ts` and offer currency selection where the schema supports it.

## 5. Mobile Audit

### 5.1 Architecture

[mobile/](../../mobile/) — React Native 0.76 + Expo 52, EAS for builds. Stack:

- **Navigation:** React Navigation 6 (stack + bottom tabs)
- **State:** TanStack Query v5 with `AsyncStorage` persister (24-hour `maxAge`)
- **Auth:** `amazon-cognito-identity-js` + `expo-local-authentication` for biometric + `expo-secure-store` for credential cache
- **Network:** `@react-native-community/netinfo` monitors state; mutations queue in AsyncStorage when offline
- **Push:** Expo Push Notifications via `notifications.ts`
- **Bundle ID:** `com.financiar.app`, version `1.0.3`, versionCode `6` per [mobile/app.json](../../mobile/app.json)
- **EAS:** dev / preview / production tracks; production goes to **Google Play internal track** only — no Apple App Store submission yet

### 5.2 Screen inventory (22 screens, matches PRD)

Auth (3) + Onboarding (1) + Main tabs (5) + More-stack (13) — see [mobile/src/screens/](../../mobile/src/screens/).

### 5.3 Mobile-specific findings

| Finding | Severity | Notes |
|---|---|---|
| **No jailbreak / root detection** | HIGH | Financial app on rooted/jailbroken device → tokens, push registration, cached transaction history all readable. Add `react-native-jail-monkey` or Expo equivalent. ([AUD-MO-001](#7-risk-register)) |
| **No certificate pinning** | HIGH | API traffic is vulnerable to MITM via compromised device CA store or rogue corporate proxy. ([AUD-MO-002](#7-risk-register)) |
| **AsyncStorage is unencrypted** | HIGH | TanStack Query persister puts every cached API response (including transaction history, balances, push tokens) into plaintext. Move sensitive items to Keychain/Keystore via `expo-secure-store`. ([AUD-MO-003](#7-risk-register)) |
| **Biometric not re-prompted per financial action** | MEDIUM | One-time Face ID at app start unlocks all subsequent transactions until session ends. Industry standard is biometric re-prompt for high-value or all financial mutations. ([AUD-MO-004](#7-risk-register)) |
| **Push notifications: no device segmentation** | LOW | One Expo channel for everything → cannot turn off only marketing notifications, etc. |
| **Offline reads not implemented** | MEDIUM | Mutation queue exists; no offline-first read strategy with SQLite or local index. UI shows blank when offline. |
| **No crash reporting** | MEDIUM | No Sentry/Bugsnag. Production-side mobile errors invisible. |

### 5.4 Build & release configuration

- `app.json` declares `USE_BIOMETRIC` and `USE_FINGERPRINT` Android permissions; correct
- `targetSdkVersion: 35`, `minSdkVersion: 24` — current
- `eas.json`: dev (internal client), preview (APK, internal), production (App Bundle, Google Play internal track)
- Expo OTA updates are not explicitly disabled — for a fintech app this is a side-channel-update risk if the EAS account is ever compromised. Consider pinning to specific runtime versions and disabling OTA in production builds.

### 6.2 CI/CD pipeline

[.github/workflows/deploy.yml](../../.github/workflows/deploy.yml) — total 69 lines; jobs:

1. Checkout
2. Configure AWS via OIDC
3. ECR login
4. Docker build + push (tags `<sha>` and `latest`)
5. `aws ecs update-service --force-new-deployment`
6. `aws ecs wait services-stable` (10-minute timeout)

**Conspicuously absent:**

- No `npm ci`
- No `npm test`
- No `npm run check` (TypeScript compile)
- No `npx playwright test`
- No `npm audit` / `osv-scanner` / Trivy scan of the Docker image
- No production-deploy approval gate

A broken commit ships to production unless the engineer ran tests locally and remembered. ([AUD-IN-001](#7-risk-register))

### 6.3 Dockerfile

- Multi-stage: builder (Node 22 Alpine, npm 11 via tarball fallback, full `npm ci`, build) → production (Node 22 Alpine, copy `dist`/`migrations`/`scripts`, non-root `appuser` UID 1001, healthcheck on `/api/health`, `CMD: run-migration.cjs migrate && node dist/index.cjs`)
- Production image still includes dev dependencies — `npm ci --omit=dev` not used. Bundle bloat is moderate (~ tens of MB), not a security issue per se but a cleanliness one.
- The `npm@11` install fallback (lines 8–14) papers over the absence of npm 11 in Alpine repos. Pinning Node 22 image with bundled npm 11 (e.g. `node:22-alpine` once it ships npm 11) will simplify this. The recent commit history is dominated by Dockerfile npm fixes.

### 6.4 `.env.example`

Reviewed: contains placeholders (`us-east-1_XXXXXXXXX`, `your-cognito-client-id`, `sk_test_your_paystack_secret_key`, etc.) — no real production secrets leaked here. Earlier reports claiming leaked AWS account IDs were inaccurate; the only AWS-account reference visible is in `.github/workflows/deploy.yml` via the `AWS_DEPLOY_ROLE_ARN` secret (which is the right pattern).
